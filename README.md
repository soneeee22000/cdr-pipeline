# cdr-pipeline

Event-driven **CDR (Call Detail Record)** ingestion, rating, and reconciliation pipeline. Built as a realistic mini-simulation of what an MVNO aggregator's back-end does: capture every billable mobile event (SMS, voice, data) from upstream operators, apply the right tariff, persist both the raw record (MongoDB) and the rated record (MySQL), and expose a reconciliation API for MVNOs.

**Stack:** Java 21 · Spring Boot 3.5 · Spring Kafka · Spring Data JPA (MySQL) · Spring Data MongoDB · Maven · Docker · JUnit 5 · Testcontainers

## Why two databases?

- **MongoDB** stores the **raw** CDR stream, exactly as received. Schema-flexible because upstream operators change payload shapes and we don't want ingestion to block on schema migrations. This is also the legal "source of truth" for audit / replay.
- **MySQL** stores the **rated** CDR stream — normalized, indexed by `(mvno_id, occurred_at)` and `(subscriber_id)` for fast reconciliation queries and billing aggregation.

This split (raw NoSQL + rated SQL) is a standard telecom back-end pattern.

## Architecture

```
 ┌──────────────┐   Kafka   ┌────────────────────┐    ┌──────────────┐
 │  CDR source  │──cdr.events──▶│  CdrKafkaConsumer  │──▶│  Mongo raw   │
 └──────────────┘           │   (group=cdr-pipe)  │    └──────────────┘
                            └────────┬───────────┘
                                     ▼
                            ┌─────────────────────┐     ┌──────────────┐
                            │   RatingService     │────▶│  MySQL rated │
                            │  (tariff lookup +   │     └──────────────┘
                            │   idempotent save)  │
                            └─────────────────────┘
                                     ▲
                            ┌─────────────────────┐
                            │  ReconciliationAPI  │
                            │  GET /api/...       │
                            └─────────────────────┘
```

## Idempotency

`RatingService.rateAndStore` checks `ratedCdrRepository.existsById(eventId)` before inserting, so a replayed Kafka offset does not double-charge. The `eventId` is the natural key generated by the upstream event source.

## Run it

Prerequisites: Java 21, Docker, Maven wrapper is included.

```bash
docker compose up -d
./mvnw spring-boot:run
```

Publish a test event (see `scripts/produce-sample.sh`), then:

```bash
curl "http://localhost:8080/api/reconciliation/mvno-acme?from=2026-04-01T00:00:00Z&to=2026-05-01T00:00:00Z"
```

## Tests

Integration tests use **Testcontainers** (real MySQL + real Mongo + real Kafka in Docker) for honest coverage. Cucumber feature files document reconciliation behavior as executable specifications.

```bash
./mvnw test
```

## Status

Portfolio project. Not production. Tariff catalog is seeded locally; no upstream operator signaling is real. Built to demonstrate Spring Boot idiomatic structure, Kafka consumer semantics, and the raw-vs-rated telecom-billing split.
